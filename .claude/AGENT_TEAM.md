# Agent Team Configuration

This document defines the specialized agent team for the Local Business CMS Framework and automatic workflow routing rules.

## üéØ Core Team (Essential - Always Active)

### 1. **Sanity CMS Master** (`sanity-cms-master-doctrine`)
**Primary Responsibility**: All Sanity CMS operations, schema management, GROQ queries, multi-tenant dataset architecture

**Auto-Route When**:
- Schema modifications or additions
- GROQ query optimization needed
- Dataset cloning or migrations
- Type generation issues
- Content modeling questions
- Multi-tenant data isolation tasks
- Sanity Studio customization

**Example Triggers**:
- "Add a new field to the service schema"
- "Optimize this GROQ query"
- "Deploy schema to all datasets"
- "Fix type generation errors"

## ü§ñ Automatic Routing Rules

### Rule Priority System

**Level 1 - Exact Match (Highest Priority)**
```
Keywords ‚Üí Agent Mapping:
- "schema", "GROQ", "dataset" ‚Üí sanity-cms-master
- "component", "Next.js", "performance" ‚Üí frontend-developer-pro
- "test", "Playwright", "failing" ‚Üí test-writer-fixer
- "benchmark", "Core Web Vitals", "LCP" ‚Üí performance-benchmarker
- "deploy", "CI/CD", "Vercel" ‚Üí devops-automator
```

**Level 2 - Context Match**
```
File Types ‚Üí Agent:
- src/sanity/schemaTypes/** ‚Üí sanity-cms-master
- src/components/** ‚Üí frontend-developer-pro
- tests/** ‚Üí test-writer-fixer
- src/app/**/page.tsx ‚Üí frontend-developer-pro
- .github/workflows/** ‚Üí devops-automator
```

**Level 3 - Task Type Match**
```
Task Categories:
- Content Modeling ‚Üí sanity-cms-master
- UI Development ‚Üí frontend-developer-pro
- Quality Assurance ‚Üí test-writer-fixer
- Performance ‚Üí performance-benchmarker
- Infrastructure ‚Üí devops-automator
```

## üìã Common Workflow Patterns

### Pattern 1: New Feature Implementation
```
1. sanity-cms-master ‚Üí Define schema
2. frontend-developer-pro ‚Üí Build UI components
3. test-writer-fixer ‚Üí Write comprehensive tests
4. performance-benchmarker ‚Üí Verify performance
5. devops-automator ‚Üí Deploy to production
```

### Pattern 2: Bug Fix
```
1. test-writer-fixer ‚Üí Reproduce with test
2. frontend-developer-pro OR backend-architect ‚Üí Fix bug
3. test-writer-fixer ‚Üí Verify fix and update tests
4. devops-automator ‚Üí Deploy fix
```

### Pattern 3: Schema Update (Multi-Tenant)
```
1. sanity-cms-master ‚Üí Update schema locally
2. sanity-cms-master ‚Üí Test with one dataset
3. devops-automator ‚Üí Deploy to all 100+ datasets
4. test-writer-fixer ‚Üí Validate across environments
```

### Pattern 4: Performance Optimization
```
1. performance-benchmarker ‚Üí Identify bottlenecks
2. frontend-developer-pro ‚Üí Implement optimizations
3. test-writer-fixer ‚Üí Ensure no regressions
4. performance-benchmarker ‚Üí Re-benchmark
5. devops-automator ‚Üí Deploy if improved
```

### Pattern 5: Multi-Site Launch
```
1. project-shipper ‚Üí Coordinate launch plan
2. sanity-cms-master ‚Üí Clone datasets
3. frontend-developer-pro ‚Üí Verify site rendering
4. test-writer-fixer ‚Üí Run full test suite
5. devops-automator ‚Üí Deploy to new sites
6. experiment-tracker ‚Üí Monitor rollout
```
